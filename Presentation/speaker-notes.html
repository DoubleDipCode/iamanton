Slide 05 — Design Decisions
Your two strongest points

Decision 01 (Modular, not monolithic): "We tried a single model with all instructions. It couldn't maintain coherence — instructions bled across tasks. The persona mapper would start using KPI language. So we isolated each step into its own agent with its own system message. It was the only way to get consistent output."

Decision 02 (Database as output channel): "When we passed sub-agent outputs back through the orchestrator, it would summarize or rewrite them. The user never got the real output. So we had sub-agents write directly to the database. The orchestrator only gets a completion signal — never the content itself. This solved the lossy summarization problem."

Why this matters to Kevin
These are real engineering insights discovered through failure. Most strategist candidates from McKinsey can't talk about LLM failure modes because they haven't built anything. You can. This is your differentiator.

Likely Questions from Kevin
Prepared responses
"What was the hardest part?"
Getting agents to follow instructions consistently. We rewrote system messages dozens of times. The breakthrough was isolation — one agent per task, own system message, own context. And the database-bypass pattern for outputs, because chained LLM calls corrupt content.
"How do you evaluate which model to use?"
Empirically. We test each model on the actual task with real briefs. For sub-agents, the key metric was tool-call reliability — not generation quality. Grok was the most consistent at invoking our logging tools correctly, and the speed advantage was significant at 11 sequential agents.
"What happens when agents disagree? Like the assumption logger flags something the product truth report contradicts?"
They don't directly interact — each agent writes its output independently. But downstream agents read all prior outputs, so contradictions surface naturally. The 4Ps Summary agent (Step 6) acts as a synthesis/quality gate — it has the strictest language rules and forces everything into a coherent summary. If there's a real contradiction, the human catches it there. Honestly, this is an area for improvement — I'd want automated consistency checks.
"How does this compare to what Sierra does?"
The architectural instincts converge — modular agents, specialized models, supervisors enforcing guardrails, persistent state. The gap is scale. My system handles internal creative workflows. Sierra handles millions of customer-facing interactions in regulated industries. But the design patterns translate: decomposition, specialization, persistent state, guardrails. I arrived at these independently, which means I'll understand Sierra's system deeply from day one.
"Why Sierra? Why this role?"
I've spent 9 months building multi-agent systems — not reading about them, building them. Sierra is the company doing this at the highest level in enterprise. The strategist role combines the analytical thinking I did in partnerships at Shakr with the agent design work I'm doing now. I want to work on agent systems where the stakes are real — customer-facing, regulated, millions of interactions. That's where this gets interesting.
