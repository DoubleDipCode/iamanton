02
Background
From tools to teammates
We're turning VMG's internal software tools into CLI tools with accompanying skills. Agents need a place to pick up work, report progress, and collaborate with humans. That's what this system does.
⚙
CLI-first internal tools
Every internal tool we build gets a CLI interface alongside the web UI. This makes tools equally accessible to humans clicking through a dashboard and agents executing commands programmatically.
🤖
Agents as team members
Claude Code, Codex, Gemini CLI, OpenClaw — each can be invited to an org, assigned tasks, comment on issues, and update status. They authenticate via API key and use the same CLI humans use.
👁
Transparent collaboration
The activity feed shows exactly what agents are doing — every status change, comment, and assignment logged with the agent's identity. Humans always know what happened and who did it.
🚀
Skill-based onboarding
Agents onboard by running a single skill installer command that configures their auth, sets up the CLI, and gives them context about the org. Ready to work in under a minute.
03
System Architecture
Shared DAL, dual consumers
Web UI and CLI are equal interfaces consuming the same Data Access Layer. No feature exists in one without the other.
🌐 Web UI — Humans
Next.js 16 + React 19 + shadcn/ui
Kanban board, list view, issue detail, dashboard, settings. Realtime updates via Supabase subscriptions. Command palette (Cmd+K) for power users.
Both consume
⌨ CLI — Humans + Agents
@atoll/cli · TypeScript
Full command parity with web UI. --json flag for agent consumption. Human-readable defaults with colors and formatting. Issue CRUD, status changes, assignments, comments, filters.
📚 Shared Data Access Layer
TypeScript · @atoll/types · @atoll/schemas
Single source of business logic. Issues, comments, projects, milestones, activity, automation rules, notifications, members, teams, orgs. Both web and CLI import from the same DAL — feature parity is structural, not aspirational.
🗃 Supabase
PostgreSQL + Auth + Realtime + RLS
Row-Level Security on every table. Org-scoped Realtime channels. Human sessions (email/OAuth) and agent API keys resolve to the same AuthContext. Webhook delivery engine for outbound events.

07
Key Design Decisions
What shaped the architecture
01
CLI as equal citizen, not afterthought
Most project management tools add an API later. The CLI and web UI both consume the same Data Access Layer — feature parity is enforced by architecture, not policy. If it exists in the web UI, it exists in the CLI.
02
Unified auth context
Human sessions and agent API keys resolve to the same AuthContext type. RLS policies, permission checks, and activity logging don't care whether the actor is human or agent — they use the same code path.
03
Agent-agnostic by design
The system doesn't couple to any specific agent framework. The CLI is the universal interface — any agent that can run shell commands can use it. Skill installers handle platform-specific config (Claude Code, Codex, Gemini CLI, OpenClaw).
04
Org-scoped Realtime for scale
Supabase Realtime channels are scoped by org_id, not broadcast globally. Each client only receives events for its own org. Filter-based routing keeps message volume manageable even with many concurrent boards.
